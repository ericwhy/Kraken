# Kraken vs. Importer Persistence Layer Gap Analysis

We need to perform an analysis that will determine what will be required to bring the persistence layer generated by the **Kraken** project up to par with that generated by the **Importer** project.

## Kraken Project
The goal of Kraken is to migrate the Kore Commerce product from .NET Framework to the latest version of .NET Core.  Additionally, it will replace the existing persistence layer with one based on Entity Framework.  It is currently in a shelved state, but we wish to evaluate how much effort would be needed to give it the same level of EF-based persistence capability that the Importer Project has produced.

## Importer Project
The Importer project's goal is to replace the existing Javascript- and SQL-based data import functionality for Kore Commerce with a modernized version that uses Entity Framework and REST APIs.  It includes a code-generated persistence layer that is capable of handling complex data import scenarios such as mapping flat JSON records across multiple tables, looking up external key values from the API call and mapping them to the proper internal database IDs, and handling large batch updates.

## Architecture Similarities
Both of these projects have a service-based persistence layer that isolates database code from the rest of the application.  They employ metadata-driven code generation to create these persistence layers.  The Importer project's persistence service converts import objects, which are created by the API, to EF entities.  The Kraken project generates its base business objects, in addition to EF entities, as part of the persistence layer and these are what are passed through its persistence API. 

### kamlbo
These code generators are driven by metadata stored in *kamlbo* files.  kamlbo is an XML format that describes business objects, their properties and relationships, among other things.  There are actually two flavors of kamlbo: one which is used by the existing Kore Commerce codebase, and another called *Kraken kamlbo*, which is a simplified version of the same metadata.  Both the Kraken and Importer projects operate off of Kraken kamlbo.

### Code Generators
Each project has its own code generator.  These generators use an object model representation of the metadata, which is loaded from kamlbo files at runtime.  The code generators then traverse the objects in the model to determine what code to generate and how.

### State of the Projects
At this point in time, the Importer's code generation capabilities are considerably greater than those of Kraken.  While the Importer's kamlbo model started off the same as the Kraken one, it has grown in capabilities since then.  It is also enhanced by a separate API object model which is driven by a different metadata file called *kamlapi*.  It is expected that Kraken will not make use of kamlapi.  There will be a need to bring the current Importer kamlbo object model back into the Kraken project, though it's unclear whether the two projects will eventually share the same model classes.  This would be preferable.

## Desired Analysis
We wish to know what capabilities are in each project's code generator and persistence layer, and how they compare.  The goal is to understand how much and what work will be needed to have Kraken support persistence of all of the object structures that are currently supported by the Importer.  In addition, we would like to know what aspects of kamlbo are not currently supported by either project.  This includes object structures that could be expressed in kamlbo but are not in found in the sample kamlbo you are provided.

# Gap Analysis Results

## Executive Summary

The Importer project has significantly more advanced persistence capabilities than Kraken. Bringing Kraken up to parity would require substantial work across three main areas:

1. **KamlBo Model Classes** - 8 new/enhanced classes needed
2. **Code Generators** - 4 new generators + enhancements to 4 existing ones
3. **Generated Persistence Layer** - Support for ~12 additional features

**Estimated effort**: Medium-to-Large (likely several weeks of development work)

## 4. KamlAPI (Importer Only)

The Importer uses a second metadata format, **kamlapi**, which defines:
- Which entities are involved in an import
- API parameter-to-property mappings
- External property definitions (XRef IDs)
- External FK mappings
- Root entity identification

**Kraken does not need kamlapi** since it generates business objects for direct application use rather than import APIs. However, some kamlapi concepts may be useful:
- External key definitions could move to kamlbo
- FK mapping definitions could be expressed in kamlbo relations

## 6. Recommended Migration Path

### Phase 1: Model Parity (Foundation)
1. **Port KamlBo model classes from Importer to Kraken**
   - `KamlBoTable`, `KamlBoCompositeData`, `KamlBoTableRelation`, `KamlBoTableKeyMap`, `KamlBoCriteria`
   - `KamlBoEntityList`, `KamlBoModel`, `ICrossDomain`
   - `Key`, `IValidatable`, `ValidationErrorList`
   - Enhanced `KamlBoDomain`, `KamlBoEntity`, `KamlBoEntityProperty`, `KamlBoEntityRelation`

2. **Implement post-parse processing**
   - `AfterParseProcessing()` for resolving entity references
   - `CreateMissingRelations()` for bidirectional relationships

### Phase 2: Entity Support Enhancements
3. **Add inheritance support**
   - `SuperType`/`SuperDomain` parsing and property inheritance
   - Inherited properties and relations

4. **Add composite table support**
   - `CompositeData` parsing
   - Multi-table entity generation

5. **Add soft delete support**
   - `DeletedField`/`DeletedValue` parsing
   - Soft delete in repositories and services

6. **Add partition support**
   - `PartitionProperty`/`PartitionValue` parsing
   - Partition filtering in queries

### Phase 3: External Key Support (Optional for Kraken)
7. **Add external key support** (if needed)
   - `IsExternal` property flag
   - `InternalValuePropertyName` mapping
   - `FindByExternalKeyAsync()` repository method

### Phase 4: Generator Enhancements
8. **Enhance existing generators**
   - `EntityFileGenerator`: inheritance, composites
   - `ConfigurationFileGenerator`: composite relationships
   - `ContextFileGenerator`: scope functions
   - `RepositoryFileGenerator`: soft delete, external keys
   - `ServiceFileGenerator`: UpdateOrInsert pattern
